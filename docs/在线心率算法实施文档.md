# 在线心率算法实施文档

> 本文档记录在线心率估计算法从 MATLAB 离线版移植到 STM32L452CEU6 在线版的完整上下文，供后续开发和调试参考。

---

## 1. 项目概述

### 1.1 目标
将 MATLAB 离线心率解算算法 (PPG + LMS 自适应滤波 + FFT 融合) 部署到 STM32L452CEU6 单片机，实现 125Hz 采样、1Hz 心率输出的实时在线运行。

### 1.1.1 项目核心定位
本项目以 **HF 桥顶信号 (Ut1, Ut2) 作为自适应滤波参考信号**为核心路径，优先保证 LMS-HF 路径的心率估计质量。具体策略：

| 状态 | 融合选择 | 说明 |
|------|---------|------|
| **运动段** | **LMS-HF 路径** (主) | HF 桥顶信号与 PPG 的运动伪影高度相关，自适应滤波消除效果最优 |
| 静息段 | Pure-FFT 路径 | 信号干净时 FFT 精度最高 |
| LMS-ACC 路径 | 底线参考 | 始终计算，作为对比参考，不参与融合决策 |
| Pure-FFT 路径 | 底线参考 (静息主选) | 运动段作为参考，静息段作为主选 |

**后续算法优化应重点围绕 LMS-HF 路径展开**，包括 HF 信号质量评估、LMS 阶数/步长调优、频谱惩罚策略等。

### 1.2 硬件平台
| 参数 | 值 |
|------|-----|
| MCU | STM32L452CEU6 |
| 内核 | ARM Cortex-M4F (带单精度 FPU) |
| 主频 | 80 MHz |
| RAM | 160 KB |
| Flash | 512 KB |
| 采样率 | 125 Hz (TIM16 定时器硬件固定) |
| 数据通道 | PPG (MAX30101 绿光) + ACC (MIMU LSM6DS3, SPI2) + HF/ADC (ADS124S06, SPI1, 4 通道桥臂) |
| 通信 | UART2 @ 115200 bps (蓝牙透传) |

### 1.3 传感器映射
| 信号 | 传感器 | 接口 | 数据格式 |
|------|--------|------|----------|
| PPG | MAX30101 (绿光 LED) | I2C (PA0/PA1) | 18-bit, 1000sps/4xavg = ~250 FIFO/s |
| ACC X/Y/Z | MIMU LSM6DS3TR-C | SPI2 (PB7/PB8 CS) | 16-bit signed, +/-4g |
| HF (桥臂) | ADS124S06 | SPI1 (PA4 NSS) | 24-bit signed, 4 通道 |
| 角速度 | LSM6DS3TR-C (陀螺) | SPI2 | 16-bit (算法未使用) |

### 1.4 当前工作模式
`CURRENT_WORK_MODE = MODE_HEART_RATE` (绿光 PPG 心率模式)

## 2. 算法架构

### 2.1 整体流水线

```
TIM16 ISR (125Hz)
    |
    v
START_CONV pulse --> ADC DRDY ISR
    |                       |
    v                       v
ADC 4通道读取          ACC_6BytesRead()
    |                       |
    +--------+--------+------+
             |
             v
    Main Loop: ADC_1to4Voltage_flag == 4
             |
             +-- PPG FIFO 批量读取 + 均值 --> float ppg_val
             +-- ACC 16-bit --> float ax, ay, az
             +-- ADC/HF 16-bit --> float hf_val
             |
             v
    HR_PushSample() --> buf_1s_*[125] (每 tick 1 个 float)
             |
             v (当 sample_idx >= 125)
    flag_1s_ready = 1
             |
             v
    HR_RunSolver() [每秒执行一次]
             |
    1. 原始窗口平移 (memmove + memcpy, 5 通道)
    2. IIR 流式滤波 (仅新 125 点, 状态连续)
    3. 滤波窗口平移
    4. 运动检测 (ACC 幅值 std vs 阈值)
    5. 窗口填充检查 (需 >= 8 秒)
    6. 时延对齐 (去均值点积, +/-5 采样点)
    7. 路径A: LMS-HF (2级级联) + FFT + 频谱惩罚
    8. 路径B: LMS-ACC (3级级联) + FFT + 频谱惩罚
    9. 路径C: 纯FFT + Hamming窗 + 频谱惩罚
   10. 融合: 运动→LMS-HF, 静息→FFT (中值平滑跟随融合路径)
   11. 变化率限制
   12. 输出: hr_bpm (float)
             |
             v
    21字节心率结果包 --> UART DMA --> 蓝牙 (仅 ENABLE_RAW_DATA_PACKET=0)
    21字节原始数据包 --> UART DMA --> 蓝牙 (仅 ENABLE_RAW_DATA_PACKET=1, 互斥)
```

### 2.2 MATLAB 算法到 C 的关键映射

| MATLAB 函数 | C 实现 | CMSIS-DSP API |
|-------------|--------|---------------|
| `filtfilt` (零相位双向滤波) | 单向流式 IIR | `arm_biquad_cascade_df1_f32` |
| `fft(signal, 8192)` 复数 FFT | 4096 点实数 RFFT | `arm_rfft_fast_f32` |
| `corr(ppg, ref)` 皮尔逊相关 | 去均值点积近似 | `arm_dot_prod_f32` |
| 自定义 NLMS (`lmsFunc_h.m`) | 归一化 LMS | `arm_lms_norm_f32` |
| `findpeaks` | 手动局部极大值检测 | 无 |
| `sort(amps, 'descend')` | 插入排序 | 无 |
| `smoothdata(..., 'movmedian', 5)` | 环形缓冲区中值 | 无 |

## 3. 设计决策记录

### 3.1 三路径融合策略: HF 为主，ACC/FFT 为底线参考
- **结论**: 保留全部 3 条路径。LMS-HF (桥顶信号参考) 为运动段主选路径，LMS-ACC 和 FFT 始终计算作为底线对比参考。预估算力 ~9ms/秒 (仅 1% CPU), 实时性无压力。
- **核心**: HF 桥顶信号 (Ut1, Ut2) 与 PPG 运动伪影的相关性优于 ACC，运动段自适应滤波消除效果更佳。
- **文件**: 已在算法初始化中分配了 HF (2级) 和 ACC (3级) 的完整 LMS 状态。

### 3.2 运动阈值: 混合方案
- **讨论**: 每次自动标定 vs 固定阈值？
- **结论**: 采用混合方案:
  1. 开机使用硬编码默认值 `Default_Motion_Th = 200.0f` (立即可用)
  2. 后台自动标定: 收集 8 个静息窗口的 ACC std 均值, 计算 `Motion_Th_Scale * baseline_std` 覆盖默认值
  3. 标定期间不阻塞心率输出, 使用默认阈值判断运动
- **风险**: 默认值需要根据实际硬件调试调整。不同佩戴方式可能影响基线。

### 3.3 数据发送模式 (互斥)
- **讨论**: 继续发送 125Hz 原始数据还是只发心率？
- **结论**: 两种模式互斥, 通过 `ENABLE_RAW_DATA_PACKET` 编译开关切换:
  - `0` (默认): 在线心率模式, 运行算法, 仅发送 1Hz 心率结果包 (0xAA 0xCC)
  - `1`: 原始数据模式, 不运行算法, 仅发送 125Hz 原始数据包 (0xAA 0xBB)
  - 心率包包含: 融合BPM/运动标志/3路径BPM/PPG信号强度/校准状态/时间戳/XOR校验
  - 互斥发送避免了两种数据包在 UART DMA 上冲突

### 3.4 IIR 滤波: 流式策略
- **关键**: 不能对 8 秒窗口重新滤波 (IIR 是因果滤波器)
- **方案**: 仅对新增 125 个样本滤波, 状态在 biquad 实例中自然传递
- **代价**: 需要维护独立的 `filt_*` 滤波后窗口 (额外 20KB RAM)

### 3.5 FFT 精度权衡
- **MATLAB**: 8192 点复数 FFT (分辨率 0.015 Hz = 0.9 BPM)
- **C**: 4096 点实数 RFFT (分辨率 0.031 Hz = 1.8 BPM)
- **评估**: 1.8 BPM 分辨率对心率检测足够, 节省一半内存

## 6. 关键技术细节

### 6.1 IIR 带通滤波器
- **类型**: 4 阶 Butterworth (2 个 biquad 节)
- **频率**: 0.5 - 5 Hz (对应 30-300 BPM)
- **系数来源**: `scipy.signal.butter(4, [0.5, 5], btype='bandpass', fs=125, output='sos')`
- **CMSIS-DSP 格式**: `[b0, b1, b2, -a1, -a2]` 每节 5 个系数
- **流式要求**: 每秒仅滤波新 125 样本, biquad 状态跨调用保持

### 6.2 LMS 自适应滤波
- **CMSIS-DSP API**: `arm_lms_norm_f32`
- **参数映射**: pSrc=参考噪声(ACC/HF), pRef=期望信号(PPG), pErr=去噪输出
- **级联**: HF 路径 2 级, ACC 路径 3 级
- **双缓冲区**: 级联使用 tmp/tmp2 交替, 避免读写冲突
- **自适应步长**: `mu = LMS_Mu_Base - corr/100` (corr 为相关系数)
- **自适应阶数**: `order = min(Max_Order, max(1, floor(|delay| * factor)))`

### 6.3 时延搜索 (ChooseDelay)
- **范围**: d = -5 到 +5 采样点
- **方法**: 对 PPG 和参考信号去均值后计算点积 (近似皮尔逊相关)
- **简化**: 相比 MATLAB 的 `corr()`, 在线版用去均值点积替代, 减少计算量
- **ref 缓冲区偏移**: ref 数组从索引 5 开始对齐, ±5 偏移通过调整起始索引实现

### 6.4 zscore 归一化
- **位置**: LMS 调用前, 在 scratch_a/b 上操作 (不污染 filt_* 数据)
- **步骤**: arm_mean → arm_offset(减均值) → arm_std → arm_scale(除标准差)
- **保护**: std < 1e-6 时钳制, 防止除零

### 6.5 频谱惩罚
- **目的**: 抑制运动频率及其二次谐波对心率检测的干扰
- **方法**: 参考 (ACC/HF) 信号 FFT 主峰 = 运动频率; PPG 峰值中接近运动频率的幅值乘以惩罚权重
- **参数**: 惩罚宽度 0.2 Hz, 惩罚权重 0.2

## 7. 数据输出协议

### 7.0 当前 Raw 数据包 (35 字节, 100Hz)

2026-04-24 起, 原始数据模式使用 35 字节 Raw 包, 在原 33 字节多光谱包基础上追加固件侧采样序号:

```
0-1    帧头                0xAA, 0xBB
2-9    ADS124S06桥压        4路uint16 BE
10-15  ACC X/Y/Z           int16 BE
16-21  GYRO X/Y/Z          int16 BE
22-30  PPG Green/Red/IR    3 bytes each, 17-bit原始值
31-32  Seq                 uint16 BE, Raw采样序号
33     XOR                 bytes[2..32]异或
34     帧尾                0xCC
```

`Seq` 在固件每生成一个 Raw 采样周期后递增, 即使 `HAL_UART_Transmit_DMA` 忙或返回错误也递增。上位机据此统计真实链路缺口: `missing_count`、`expected_count`、`Loss`、`RX Hz` 和 `DEV Hz`。

### 7.0.1 Raw 链路诊断 STATUS 包 (53 字节, 1Hz)

2026-04-30 起, 原始数据模式增加第一阶段链路诊断 STATUS 包。该包不替代 35 字节 Raw 数据包, 仅用于采集后定位丢失来源。

```
0-1    帧头                         0xAA, 0xDD
2      protocol_version             uint8, 当前为1
3-6    mcu_time_ms                  uint32 BE, HAL_GetTick()
7-10   sample_counter               uint32 BE, TIM16采样tick总数
11-14  adc_drdy_counter             uint32 BE, ADC DRDY中断次数
15-18  frame_counter                uint32 BE, Raw DATA组帧次数
19-22  tx_start_counter             uint32 BE, Raw DATA UART DMA启动成功次数
23-26  tx_done_counter              uint32 BE, Raw DATA UART DMA完成回调次数
27-30  tx_busy_counter              uint32 BE, Raw DATA发送时HAL_BUSY次数
31-34  tx_error_counter             uint32 BE, Raw DATA UART/DMA错误次数
35-38  adc_error_counter            uint32 BE, ADC状态异常次数
39-42  imu_error_counter            uint32 BE, IMU运行期异常次数
43-46  ppg_fifo_empty_counter       uint32 BE, PPG FIFO空读次数
47-50  ppg_fifo_overflow_counter    uint32 BE, PPG FIFO接近满/溢出风险次数
51     XOR                          bytes[2..50]异或
52     帧尾                         0xCC
```

固件端只统计 Raw DATA 发送路径的 `tx_start_counter`、`tx_done_counter`、`tx_busy_counter` 和 `tx_error_counter`; STATUS 自身发送不计入这些 Raw 发送计数。上位机录制时生成同名 `_status.csv`, 用于和 Raw CSV 一起分析 `sample_counter -> frame_counter -> tx_done_counter -> PC received` 的链路守恒关系。

## 8. 默认参数表

| 参数 | 默认值 | 说明 |
|------|--------|------|
| LMS_Mu_Base | 0.01 | LMS 基础步长 |
| Spec_Penalty_Width | 0.2 Hz | 频谱惩罚宽度 |
| Spec_Penalty_Weight | 0.2 | 频谱惩罚权重 (越小抑制越强) |
| HR_Range_Hz | 0.25 Hz | 运动段心率搜索范围 |
| HR_Range_Rest_Hz | 0.4 Hz | 静息段心率搜索范围 |
| Slew_Limit_BPM | 10.0 BPM/s | 运动段心率变化率上限 |
| Slew_Step_BPM | 2.0 BPM | 运动段心率单步步进 |
| Slew_Limit_Rest_BPM | 8.0 BPM/s | 静息段心率变化率上限 |
| Slew_Step_Rest_BPM | 1.0 BPM | 静息段心率单步步进 |
| Motion_Th_Scale | 3.0 | 运动阈值倍数 (x baseline_std) |
| Default_Motion_Th | 0.07 g | 默认运动阈值 (g 值，内部转换为 LSB) |

### 10.2 潜在优化方向 (围绕 HF 主路径)
1. **HF 信号质量评估**: 在 LMS-HF 路径入口增加参考信号质量门控, 丢弃低质量 HF 数据
2. **LMS-HF 参数调优**: 阶数 (当前 MAX_ORDER=16)、步长 (当前 Mu=0.01)、级联数 (当前 2 级) 需实机验证收敛性
3. **降采样**: 如果 125Hz 分辨率对心率检测过高, 可降采样到 25-50Hz 以减少计算量
4. **PpgPeace 信号质量评估**: 当前未集成, 可作为额外质量门控
5. **自适应 FFT 长度**: 静息时可用 2048 点, 运动时用 4096 点

### 10.3 注意事项
1. `filt_*` 缓冲区需要首尾各多 5 个采样点以支持 DSP_FindDelay 的 +/-5 偏移。当前实现中 ChooseDelay 直接在 filt_* 上操作, 偏移靠调整起始索引实现, 但存在边界风险。如果 8 秒窗口首尾数据不完整, 可能需要额外处理。
2. main.c 中 ACC 高位数据 `allData[10..12]` 仍然发送 (向后兼容), 但算法使用完整的 16-bit ACC。两者数据格式不同, 上位机需区分。
