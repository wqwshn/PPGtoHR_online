## 一、需要硬件调试后决定的问题

### ~~P0 - 运动阈值校准~~ ✅ 已完成 (2026-03-28)

- **修改前**: `Default_Motion_Th = 200.0` (量纲不明确)
- **修改后**: 添加 `HR_ACC_LSB_PER_G = 8192.0f` 常量，阈值改为 g 值单位，默认 `0.07g`（基于实测静息 0.06-0.07g）
- **位置**: `hr_algorithm.h` 添加 ACC 灵敏度常量, `hr_algorithm.c` 中阈值自动转换为 LSB
- **效果**: 配置更直观，阈值与实际物理量对应

### ~~P0 - BPF 系数验证~~ ✅ 已完成 (2026-03-28)

- **修改前**: 2 个 biquad 节，系数错误（截止频率 0.999Hz/4.001Hz，通带衰减 98dB）
- **修改后**: 4 个 biquad 节，使用 MATLAB butter() 生成的正确系数
- **验证结果**: -3dB 截止频率 0.500Hz / 5.000Hz，通带衰减 4.35dB，误差 < 0.1%
- **影响文件**: `hr_dsp.h` (系数数组 10→20), `hr_algorithm.h` (状态数组 8→16), `hr_dsp.c` (系数更新)

### P1 - LMS 收敛稳定性

- **现状**: 每秒调用 `LMS_Init` 重置 LMS 系数
- **位置**: `hr_algorithm.c:408-425` (Step 7)
- **建议**: 观察前 3-5 秒的 LMS-HF / LMS-ACC 输出是否剧烈波动。如果前几秒不稳定，需改为保持系数跨窗口连续
- **验证方法**: 上位机观察三路 BPM 输出，特别是启动后前 10 秒的变化曲线

### ~~P1 - PPG 信号范围~~ ✅ 已确认 (2026-03-28)

- **数据解算**: 已确认正确。18-bit ADC 精度完整保留（`& 0x03FFFF`），算法输入 float 均值计算正确。
- **实测范围**: PPG 均值 500-2000（相对值），符合正常佩戴下的信号强度。
- **LED 配置**: 当前 `LED3_PA = 0x71`，如需增强信号可调至 `0xFF`（最大电流）。
- **数据精度**: 算法内部使用 float 保持完整精度，无信息丢失。

### P2 - HF 参考信号有效性

- **现状**: LMS-HF 路径使用 ADS124 的 HF 通道作为运动伪影参考
- **位置**: `hr_algorithm.c:430-504` (Step 8)
- **建议**: 实测验证 HF 信号是否真的包含与 PPG 运动伪影相关的信息。如果相关性低，这条路径可以直接砍掉，节省约 30% 算力和 3KB RAM
- **验证方法**: 对比 LMS-HF 与 LMS-ACC 的 BPM 输出一致性

## 二、代码层面可直接优化

### P0 - LMS 系数每秒重置 (最大算法缺陷)

- **现状**: `LMS_Init` 每秒调用一次，所有系数清零，LMS 永远无法充分收敛
- **位置**: `hr_algorithm.c:408-425` (Step 7 中的 `LMS_Init` 调用)
- **建议**: 改为仅在首次初始化或延迟变化超过 2 个采样点时才重置系数，否则保持系数并继续自适应。收敛后的 LMS 滤波效果会显著提升
- **修改方案**:

```c
// 当前代码 (每秒重置):
LMS_Init(&state->lms_hf[i], delay, s_config.LMS_Mu_Base, HR_WIN_SAMPLES);
LMS_Init(&state->lms_acc[i], delay, s_config.LMS_Mu_Base, HR_WIN_SAMPLES);

// 建议改为 (仅首次或延迟变化时重置):
if (state->prev_delay_hf != delay || state->win_count <= HR_WIN_SEC) {
    LMS_Init(&state->lms_hf[i], delay, s_config.LMS_Mu_Base, HR_WIN_SAMPLES);
    state->prev_delay_hf = delay;
}
```

### P1 - 重复 FFT 计算

- **现状**: 频谱惩罚分支对同一信号做了两次 `DSP_FFTPeaks`
- **位置**: Step 8/9/10 中的频谱惩罚逻辑
- **流程**: 先对 LMS 输出做 FFT -> 再对参考信号做 FFT (覆盖了 peak_freqs/peak_amps) -> 又要重新对 LMS 输出做 FFT
- **建议**: 为参考信号 FFT 使用独立的峰值缓冲区，或在惩罚前缓存 LMS 输出的 FFT 结果，避免每次额外开销约 4096 点 FFT
- **预估收益**: 减少 2 次 4096 点 FFT (LMS-HF 和 LMS-ACC 各一次)

### P1 - 中值平滑目标错误

- **现状**: `DSP_MedianSmooth` 仅对 `hr_history_fft` 做平滑，但融合心率在运动时取 `hr_lms_acc`
- **位置**: `hr_algorithm.c:710`
- **建议**: 中值平滑应作用于实际融合值的历史。需增加 `hr_history_fused` 数组，记录每秒的实际融合值，对该数组做中值平滑

### P2 - sqrt 循环可用 CMSIS-DSP 加速

- **现状**: 手写 for 循环做 sqrt
- **位置**: `hr_algorithm.c:302-308`
- **建议**: 替换为 CMSIS-DSP 的 `arm_sqrt_f32` 或分段查表，利用 Cortex-M4F 硬件 FPU 的 VSQRT 指令

### P2 - ACC 原始数据包只发高 8 位

- **现状**: 21 字节原始数据包中 ACC 字段仅 3 字节 (X/Y/Z 各高字节)
- **位置**: `main.c` 数据打包逻辑
- **建议**: 算法内部使用完整 16-bit ACC，但原始数据包精度不足。如需离线分析，建议扩展协议或增加完整的 16-bit ACC 数据字段

### P3 - s_config 全局静态变量

- **现状**: 配置存在模块级静态变量中
- **位置**: `hr_algorithm.c:18`
- **建议**: 如果未来需要多实例（如同时监测多人），应将 config 指针存入 `HR_State_t` 结构体内。当前单实例无问题，优先级最低

## 三、中长期优化方向

### 降采样评估

125Hz 采样对于心率信号 (<5Hz) 而言偏高，Nyquist 只需 >10Hz。如果实测证明 25Hz 或 50Hz 就够用:

- FFT 从 4096 降到 1024/2048
- RAM 节省 16-24KB
- 算力减半
- 窗口长度不变，只是每秒处理的采样点减少

### 信号质量评估模块

MATLAB 参考代码 (`outline-HRsolver/PpgPeace.m`) 中有信号质量评估模块，在线版本尚未集成:

- 在信号质量差时抑制 HR 输出，避免误报
- 可作为融合决策的额外权重因子
- 预估额外 RAM 开销约 2KB

### 自适应 FFT 镇长度

- 静止时信号稳定，2048 点 FFT 足够 (0.061Hz 分辨率 = 3.7BPM)
- 运动时用 4096 点保持精度
- 动态切换可节省算力

### ~~上位机协议优化~~ ✅ 部分完成 (2026-03-28)

- **修改前**: 原始数据包 (125Hz) 和 HR 结果包 (1Hz) 同时发送，占用约 2.5KB/s 带宽
- **修改后**: 添加 `ENABLE_RAW_DATA_PACKET` 开关 (main.c)，默认 0 (不发送原始数据)
- **建议**: 后续可增加命令字，由上位机动态控制是否接收原始数据
